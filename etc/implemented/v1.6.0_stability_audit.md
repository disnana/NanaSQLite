# NanaSQLite v1.6.0 stability audit

## Baseline and release gates

- Baseline before v1.6 work: 1,034 passed / 12 skipped on Windows Python 3.11.
- Current v1.6 regression suite: 1,050 passed / 12 skipped, plus 8 focused niltest specifications.
- Required gates: full pytest, Ruff, mypy, focused benchmarks, wheel/sdist build, and distribution dependency inspection.

## Resolved high-priority risks

| Area | Previous risk | v1.6 resolution |
|---|---|---|
| v2 backup | Staged writes could be absent | Synchronous flush and DLQ refusal by default |
| Backup replacement | Failure could overwrite an existing destination | Same-directory temporary DB, integrity check, atomic replace |
| Live-WAL restore | Raw file copy could omit committed WAL data | SQLite online-backup snapshot before replacement |
| Restore failure | v2 engine could remain shut down | Recreate the engine and reconnect children on failure |
| Child lifecycle | Restore could create one v2 engine per child | Parent and children share one recreated engine |
| Transaction rollback | Cache and UniqueHook index could expose rolled-back values | Invalidate cache and derived hook indexes |
| Concurrent counters | Read/modify/write could lose updates | `BEGIN IMMEDIATE` atomic `increment()` / `patch()` |

## Reviewed and deferred

- FTS, generated JSON indexes, soft delete, state-machine hooks, and expiration hooks are feature work rather than stability remediation and remain targeted for v1.6.1+.
- v2 remains intentionally single-process. Atomic operations are process-safe because they write synchronously, but the general v2 write-back cache architecture is not changed into a multi-process cache.
- Backup path authorization remains the caller's trust-boundary responsibility; v1.6 strengthens integrity and replacement semantics without introducing application-specific directory policy.

## Development-only specifications

`niltest==1.2.1` is declared only in the PEP 735 `spec` dependency group and is run against deterministic helper behavior. Production modules do not import it, and CI inspects built archives to reject accidental runtime or package inclusion.
