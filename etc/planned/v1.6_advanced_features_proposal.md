# NanaSQLite 次期拡張アーキテクチャ提案書 (v1.6+)
**テーマ: SQLiteネイティブ機能の解放と、より高度で汎用的な状態・時間管理**

現在までのNanaSQLite（v1.5.0dev1）で、「NoSQLライクな辞書操作」と「データ保護（Ultimate Hooks）」の基盤は完成しました。
ライセンス管理APIなどのより高度なユースケースにおいて、**「時間を伴う状態変更」や「検索パフォーマンス」「競合防止」**を究極まで汎用化するための次期アプローチを提案します。

---

## 1. 時間と状態の汎用化（Time & State Management）

ライセンスの「有効化してから〇〇日間」や「期限切れ」といった仕様はAPI開発で頻出するため、ライブラリ側でスマートに吸収します。

### 提案A: 状態遷移制約フック (`StateMachineHook`)
値の「状態（Status）」が、許可された順序でしか変更できないようにする汎用フックです。
* **メリット:** 「期限切れ(expired)」のライセンスを誤って「未使用(pending)」に戻してしまうようなバグを、データベース層で完全に防ぎます。
* **想定API:**
  ```python
  # pending -> active -> expired の一方通行しか許さない制約
  db.add_hook(StateMachineHook(
      field="status",
      transitions={
          "pending": ["active", "cancelled"],
          "active": ["expired", "suspended"],
          "suspended": ["active", "cancelled"],
          "expired": [], # 期限切れからは遷移不可
          "cancelled": []
      }
  ))
  ```

### 提案B: アクティベーションと連動するハードTTL (`ExpirationHook` & Auto-Reaper)
現在のNanaSQLiteのTTLは「メモリキャッシュの有効期限」ですが、これを「物理データの有効期限」へと昇華させます。
* **メリット:** 「有効化(activated_at)」を記録した瞬間に、特定の計算式で「終了日時(expires_at)」を自動計算し、期限が切れたデータは取得時に自動ブロック（またはバックグラウンドで物理削除）します。
* **想定API:**
  ```python
  # activated_at が記録されたら、その30日後に expires_at を自動設定し、期限後は読み取り不可にする
  db.add_hook(ExpirationHook(
      trigger_field="activated_at",
      duration_days=30,
      expiration_field="expires_at",
      auto_delete=False # Trueなら期限切れで物理削除、FalseならSoft Delete（無効化）扱い
  ))
  ```

## 2. SQLiteネイティブ機能の透過的API化 (Deep SQLite Integration)

NanaSQLiteのデータはJSONとして保存されていますが、これを「辞書のように使いながら、SQLiteの超強力なC言語エンジンをフル活用する」ための拡張です。

### 提案C: JSONインデックスの自動生成 (`add_json_index`)
「Discord IDからライセンスを検索する」ような処理を、メモリの全探索ではなく、SQLiteのB-Treeインデックスを使って `O(1)` クラスの速度で行えるようにします。
* **仕組み:** SQLiteの **Generated Column（生成列）** 機能とネイティブ **JSON1拡張** を裏側で自動構築します。
* **想定API:**
  ```python
  # 内部で json_extract(data, '$.discord_id') を行う仮想カラムとインデックスを自動生成
  db.add_json_index("discord_id", "$.discord_id", unique=True)
  
  # ユーザーは意識せずに爆速検索が可能に
  license = db.query("discord_id = ?", ("123456789",))
  ```

### 提案D: フルテキスト検索の透過的統合 (`enable_fts`)
膨大なログやユーザー情報から、特定の部分文字列を爆速で検索する SQLite `FTS5` (Full-Text Search) 仮想テーブルを自動同期させます。
* **メリット:** Elasticsearchなどを立てることなく、ライブラリの関数を1つ呼ぶだけで高度な全文検索エンジンが手に入ります。
* **想定API:**
  ```python
  # name と memo フィールドの全文検索インデックスを構築
  db.enable_fts(fields=["name", "memo"])
  
  # "太郎" かつ "プレミアム" を含むデータを高速検索
  results = db.search_fts("太郎 AND プレミアム")
  ```

## 3. アトミック操作とデータ保護 (Atomic & Protective Operations)

APIバックエンドとして利用する際、マルチスレッドや複数リクエストによる「競合」をSQLiteレベルで防ぎます。

### 提案E: ロックフリーな部分更新とカウンター (`json_patch` / `increment`)
「APIの残り実行回数」などを減らす際、一度Python側に読み込んで(Read)、計算して、保存(Write)すると競合します。これをデータベースエンジン側に計算させます。
* **仕組み:** SQLiteの `json_set()` 等を使用して、UPDATE文だけで処理を完結させます。
* **想定API:**
  ```python
  # レートリミットやライセンス消費など、競合を気にせず安全に -1 する
  db.increment("license:user1", field="uses_left", amount=-1)
  
  # DB側で特定のフィールドだけを直接更新
  db.patch("license:user1", {"is_active": False})
  ```

### 提案F: ソフトデリートの完全サポート (`SoftDeleteHook`)
ライセンス情報や決済情報は、法律や監査の要件で「物理的に消してはいけない」ことが多々あります。
* **仕組み:** 削除操作を行うと、物理削除する代わりに `{"_deleted": true, "_deleted_at": <timestamp>}` というフラグを追加し、通常のAPI（`get`, `query`, `[key]`）からは**完全に隠蔽（存在しないように振る舞う）**させます。
* **想定API:**
  ```python
  # テーブル全体をソフトデリートモードにする
  db = NanaSQLite("data.db", soft_delete=True)
  
  del db["license:999"] # 実際には消えず、_deleted フラグが立つ
  
  # 監査などで消えたデータを見たい場合は専用パラメータを渡す
  history = db.query_all(include_deleted=True)
  ```

## まとめ

これらの提案（特に**インデックス生成（提案C）**と**アトミックカウンター（提案E）**、そして**ソフトデリート（提案F）**）を実装することで、NanaSQLiteは「単なる便利なKVS」から、**「インフラ構築が不要な、APIバックエンド専用の完全体データベース」**へと進化します。

既存の `Ultimate Hooks` アーキテクチャと組み合わせることで、「設定ファイルを数行書くだけで、Discord API認証サーバーの全バックエンドロジックが完成する」レベルの汎用性を実現できます。
